## 1. 项目背景

### 1.1 业务现状
法拉第平台目前已有**巡检计划、电站档案、客户信息**等基础数据，但这些数据散落在不同模块中，巡检管理者需要：

- 跨模块拼接信息才能掌握**某个电站的当前巡检状态**
- 缺乏**全局地图视角**，无法直观看到电站分布和巡检进展
- 巡检车辆**位置不可见**，无法判断是否到达/离开基地
- **纸质或 Excel 报表**统计里程和完成率，数据滞后 1-2 天

### 1.2 核心痛点
1. **管理者痛点**：领导出差时无法掌握一线情况，缺乏移动端/可视化入口
2. **调度员痛点**：靠电话调度车辆，不知道车辆在哪、行驶多远
3. **巡检员痛点**：不知道下一站顺序、不了解附近是否有同事
4. **客户痛点**：询问巡检进度时无法给出准确答复

### 1.3 项目目标
打造一个**实时可视化的电站巡检运营平台**，实现"**一张图看全部**"。

| 目标 | 衡量指标 |
|------|---------|
| 缩短调度决策时间 | 调度员日常找站点时间由 15min 降至 2min |
| 实时掌握车辆位置 | 车辆位置延迟 ≤ 10 秒 |
| 数据驱动运营 | 报表 T+0 自动出，无需手工统计 |
| 异常主动预警 | 上下班异常主动触达率 100% |

### 1.4 预期收益
- **运营效率**：预计节省调度人力 30%
- **客户满意度**：进度查询响应时间由小时级降至秒级
- **管理透明**：日报自动生成，领导无需催数据

## 2. 术语定义

| 术语 | 解释 |
|------|------|
| **电站 / 变电站** | 需要巡检的电力站点 (本项目核心实体) |
| **巡检计划** | 一段时间内巡检哪些电站的安排 (含有效期) |
| **巡检组** | 负责一组电站的团队，含 1 名负责人 + N 名组员 |
| **基地** | 巡检组的车辆停放/集合地点，含上下班时间配置 |
| **GPS 设备** | 车载硬件，第三方系统统一管理并上报位置 |
| **定位点** | 用户通过地址 Geocoding 得到的地图坐标点 |
| **巡检负责人** | 对应一个巡检组的责任人 |
| **项目经理** | 客户侧对接的项目责任人 |
| **GPS 双向监控** (v1.1 新增) | 平台向车辆 GPS 设备**下发指令**（如远程重启、断油、定位查询）并接收确认的能力。区别于单向位置上报 |
| **报警宽限时间** | 上班时间之后允许车辆未离开基地的容忍时长，超出则触发报警；默认 15 分钟 |
| **提前回基地报警阈值** | 下班时间之前车辆回到基地的提前量，超出则触发报警；默认 30 分钟 |
| **车辆类型 / 动力类型** (v1.1 勘误) | 原始方案中两处「车辆类型」实为不同维度：车辆类型指皮卡/轿车，动力类型指油车/电车 |
| **聚合展示** | 当地图上电站标记数量超过阈值时，自动按地理范围合并为聚合点，逐级展开 |
| **WebSocket 推送** | 服务端主动向客户端推送实时位置数据的协议，本项目 5 秒一次 |

## 3. 用户与角色

### 3.1 用户角色

| 角色 | 主要场景 | 关键诉求 |
|------|---------|---------|
| **系统管理员** | 配置基地、车辆、权限 | 录入效率、配置灵活性 |
| **项目经理** | 跟进所辖项目巡检进度 | 进度可视、报表导出 |
| **调度员** | 日常调度、异常处理 | 实时位置、报警推送 |
| **巡检组长** | 接收任务、安排组员 | 路径优化、任务清晰 |
| **公司领导** | 宏观把控 | KPI 看板、跨组对比 |


### 📍 第一阶段：变电站地图展示 (P0)

#### FR-1.1 巡检计划关联的电站展示

**用户故事**
> 作为【调度员】，我希望地图上展示当前正在进行的所有巡检计划下的电站，以便快速掌握今日工作面。

**数据查询条件**（后端逻辑，前端不展示）
- 巡检计划 `start_time ≤ 当前时间 ≤ end_time`
- 巡检计划类型 = "变电站巡检计划"

**功能规则**
| 编号 | 规则 |
|------|------|
| R1.1.1 | 仅展示**当前有效**的巡检计划关联的电站，已过期或未开始的计划对应电站不展示 |
| R1.1.2 | 同一电站被多个计划关联时，地图上只显示一个标记，hover 展示所有计划信息 |
| R1.1.3 | 标记根据电站**电压等级**显示不同颜色（35kV/110kV/220kV/500kV） |
| R1.1.4 | 站点数量 > 50 时自动启用**聚合展示**，点击聚合点逐级展开 |
| R1.1.5 | 站点数量 ≤ 50 时直接展示所有标记 |

**界面参考**: [[index.html#screen-map|原型-变电站地图]]

**验收标准**
- [ ] 当前日期在计划有效期内时，电站出现在地图上
- [ ] 当前日期超出有效期后 5 分钟内，电站自动从地图消失
- [ ] 当电站数据 ≥ 50 个时，自动聚合，地图操作流畅
- [ ] 标记颜色与电压等级一一对应

📌 **业务来源**: 原始方案 L1-L3（第一阶段·第 1 条）

#### FR-1.3 站内电站搜索筛选

**用户故事**
> 作为【项目经理】，我希望快速搜索电站，以便客户咨询时 5 秒内定位。

**搜索范围**
- 电站名称
- 客户名称
- 地址
- 户号

**功能规则**
| 编号 | 规则 |
|------|------|
| R1.3.1 | 输入框支持上述任一字段的**模糊搜索**（LIKE） |
| R1.3.2 | 输入即搜索（debounce 300ms），无需点击搜索按钮 |
| R1.3.3 | 搜索结果**实时同步**到地图标记和右侧列表 |
| R1.3.4 | 地图自动调整视野以**包含所有命中电站** |
| R1.3.5 | 搜索结果为空时显示"未找到相关电站"提示 |
| R1.3.6 | 清空输入框后恢复展示全部电站 |

**界面参考**: 原型中「⚡ 电站搜索」Tab

**验收标准**
- [ ] 输入"南山"，所有名称含"南山"的电站被筛出
- [ ] 输入"755203"，对应户号的电站被筛出
- [ ] 搜索响应时间 ≤ 500ms（1000 电站规模）
- [ ] 清空输入后 1 秒内恢复全部展示

📌 **业务来源**: 原始方案 L5（第一阶段·第 3 条：搜索电站）

#### FR-1.4 ⭐ 通过地址进行地图定位 (本期新增)

**用户故事**
> 作为【调度员】，我希望输入任意地址定位到地图，以便判断该位置附近的待巡检电站。

**功能规则**
| 编号 | 规则 |
|------|------|
| R1.4.1 | 在搜索框顶部提供**双 Tab** 切换：`⚡ 电站搜索` / `📍 地址定位` |
| R1.4.2 | 选择"地址定位"Tab 后，输入框 placeholder 变为"输入任意地址定位..." |
| R1.4.3 | 输入即调用高德 Geocoding API，返回多个候选地址 |
| R1.4.4 | 候选列表中**第一个**为最佳匹配，显示"精确匹配"徽章 |
| R1.4.5 | 点击候选地址，地图**居中**到该坐标，放置粉色定位标记 |
| R1.4.6 | 定位标记样式：粉色 📍 + 旋转虚线圆环，显著区别于电站标记 |
| R1.4.7 | 定位后**自动激活**半径搜索，默认中心点为该坐标，半径 3km |
| R1.4.8 | 显示半径范围内的电站数量，支持调节半径滑块 (0.5km - 50km) |
| R1.4.9 | 提供"清除定位"按钮，恢复默认视图 |

**输入示例**
- "深圳市福田区市民中心" → 精确定位到市民中心
- "南山科技园" → 定位到科技园区域中心
- "错误地址 xyz" → 显示"未找到该地址，请检查后重试"

**界面参考**: 原型中「📍 地址定位」Tab + 粉色定位标记

**验收标准**
- [ ] 切换 Tab 时输入框和结果列表正确切换
- [ ] 输入"深圳市福田区市民中心"，1 秒内返回 ≥ 1 个候选
- [ ] 点击候选后地图 500ms 内居中到该位置
- [ ] 地图上出现粉色定位标记，且与电站标记视觉上有明显区分
- [ ] 半径搜索自动以定位点为中心，并显示范围内电站数
- [ ] 同一地址第二次查询响应时间 ≤ 100ms (命中缓存)

📌 **业务来源**: 原始方案 L6（第一阶段·第 4 条：地址定位）

#### FR-1.5 巡检负责人快速筛选

**用户故事**
> 作为【项目经理】，我希望按巡检负责人筛选地图，以便查看某位组长的所有电站。

**功能规则**
| 编号 | 规则 |
|------|------|
| R1.5.1 | 在左侧筛选面板提供"巡检负责人"下拉选择 |
| R1.5.2 | 下拉选项**动态加载**，列出所有当前有效的负责人 |
| R1.5.3 | 选中某负责人后，地图只展示其负责的电站 |
| R1.5.4 | 支持"全部"选项，展示全部电站 |
| R1.5.5 | 与"关键词搜索"为**AND 关系**，可叠加使用 |

**验收标准**
- [ ] 下拉中仅出现当前有效计划中的负责人

### 🚨 第二阶段：车辆报警 (P1)

#### FR-6.1 报警规则

**规则 1: 上班未离开基地**
- 触发条件：当前时间 > 上班时间 + 宽限时间，且车辆仍停留在基地 200m 范围内
- 通知对象：车辆负责人 + 项目经理
- 通知方式：站内信 + 企业微信

**规则 2: 提前返回基地**
- 触发条件：当前时间 < 下班时间 - 提前告警阈值，且车辆回到基地 200m 范围
- 通知对象：车辆负责人
- 通知方式：站内信

**规则 3: GPS 离线超时**（v1.1 待运维确认）
- 触发条件：GPS 设备超过 **30 分钟**（默认，待运维确认）未上报
- 通知对象：车辆负责人
- 通知方式：站内信 + 短信
- 可配置：阈值在基地/车辆级别可独立配置（0-120 分钟）

**规则总开关**
- 基地级报警开关：关闭后该基地所有报警都不触发
- 车辆级报警开关：关闭后该车辆所有报警都不触发
- 关闭时通知管理者"已关闭报警"

> ⚠️ **方案对齐说明**：原始方案 L10 仅提及"上班未离开基地"和"提前返回基地"两类报警，本说明书中新增的「GPS 离线超时」为运维侧技术保障需求，源自 [[法拉第/原型界面/技术方案.md\|法拉第/原型界面/技术方案.md]]，与原始业务方案**不完全一一对应**，落地前需业务方确认是否纳入本期。

## 6. 非功能需求

### 6.1 性能需求

| 指标 | 要求 | 备注 |
|------|------|------|
| 地图加载时间 | 首屏 ≤ 2 秒（1000 电站规模） |  |
| 搜索响应时间 | ≤ 500 ms | 1000 电站规模 |
| 地址 Geocoding 响应 | ≤ 1 秒（首次），≤ 100ms（缓存命中） | Redis 7 天缓存 |
| 实时位置推送延迟 | ≤ 5 秒 | WebSocket 推送（v1.1 统一为 5 秒，与 1.3 项目目标 ≤10 秒对齐到实现层） |
| 报表查询响应 | ≤ 3 秒（90 天数据） | TDengine 聚合 |
| 并发支持 | ≥ 100 用户同时在线 |  |
| GPS 上报写入 TPS | ≥ 1000 条/秒 | TDengine 时序库 |
| 报警规则判定延迟 | ≤ 30 秒 | 从触发条件满足到通知发出 |

### 6.2 可用性需求

| 指标 | 要求 |
|------|------|

| 浏览器兼容 | Chrome 90+、Edge 90+、Safari 14+ |

### 6.3 安全需求

| 项 | 要求 |
|-----|------|

### 6.4 兼容性需求

- 屏幕分辨率：1280×720 至 2560×1440
